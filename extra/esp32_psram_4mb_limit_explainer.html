Why only 4 MB of the 8 MB PSRAM on ESP32-WROVER-IE N16R8 is accessible, explained through the ESP32 chip's MMU address space limits

Root cause 1 — the MMU window is 4 MB wide, fixed in silicon
The ESP32 chip's Memory Management Unit (MMU) dedicates a specific region of the 32-bit address space for external SRAM: 0x3F40_0000 to 0x3F7F_FFFF. That is exactly 4 MB (0x40_0000 bytes). This window is hardwired in the chip's die — it is not a firmware setting, not an SDK limitation, and not something Espressif could change with a software update. The WROVER-IE datasheet (Section 3.2, sourced from ESP32 Technical Reference Manual) states: "External SRAM can be mapped into CPU data memory space. Up to 4 MB can be mapped at a time."
Root cause 2 — the MMU maps by pages, not by full chip
The ESP32 MMU works by mapping 32 KB pages of external memory into the address window. Each page entry in the MMU points to a 32 KB region of the external PSRAM. The total number of page slots only covers the 4 MB window. There are no spare page table entries to reach beyond 4 MB. The upper 4 MB of the physical chip simply has no page table entry pointing to it — there is no CPU address for it to live at.
Root cause 3 — why Espressif still ships 8 MB (R8) then?
The PSRAM chip used in N16R8 variants is an 8 MB device because that is what is manufactured and available in QSPI PSRAM packages at 80 MHz. Espressif uses the same chip across product families. The ESP32-S3, for example, has a larger MMU window and can address 8 MB of PSRAM — so the same physical 8 MB chip is fully usable on ESP32-S3. On the original ESP32 (D0WD-V3), only half is reachable. The "R8" in the part number truthfully describes the physical chip, not what the ESP32 CPU can address.
Root cause 4 — SPI bus addressing adds another constraint
The PSRAM is accessed over a shared QSPI bus via SPICS1 (chip select 1, as seen in the WROVER block diagram, Figure 1). The SPI flash uses SPICS0. The ESP32's SPI controller for PSRAM uses a 24-bit address (3 bytes) for the Quad SPI protocol at 80 MHz, which naturally limits addressable space to 2²⁴ = 16 MB. So bus addressing is not the bottleneck here — the MMU window of 4 MB is the actual constraint.

Option 1 — switch to ESP32-S3 (hardware solution)
ESP32-S3 has a redesigned MMU that supports up to 32 MB of external PSRAM. With an ESP32-S3-WROOM-2 module (N16R8), all 8 MB of PSRAM is fully addressable. If your project is still in the design phase, this is the correct long-term fix. The ESP32-S3 is Espressif's recommended successor for memory-intensive applications.
Option 2 — use flash-mapped storage for the extra space (workaround)
The 16 MB flash on the WROVER-IE N16R8 is fully accessible (flash has its own larger mapping window). You can use esp_partition + LittleFS or SPIFFS to store large assets, model weights, or audio data in flash and stream them into the 4 MB PSRAM buffer as needed. This is slower than true PSRAM access but works well for buffered playback or inference pipelines.
Option 3 — custom MMU paging (extremely advanced, not practical)
In theory, you could implement a software-managed paging scheme where you swap 32 KB pages in and out of the 4 MB window manually — essentially implementing virtual memory. In practice this requires modifying ESP-IDF at a very deep level, adds significant latency on every swap, and is not supported by any official Espressif SDK. It is not a viable option for production use.
Bottom line — the 4 MB limit is a hardware design constraint
No firmware update, no ESP-IDF version, no configuration flag, and no amount of coding can change the fact that the ESP32 chip's MMU window for external SRAM is 4 MB. The only real solution for full 8 MB PSRAM access is the ESP32-S3 chip family.
